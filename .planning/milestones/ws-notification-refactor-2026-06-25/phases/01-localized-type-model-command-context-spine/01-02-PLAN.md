<objective>
Migrate the seven plugin-family commands (install, uninstall, update, reinstall,
enable, disable, list, info — enable/disable share `enable-disable.ts`) to the
command-local notification model: each gets a co-located `<cmd>.messaging.ts`
sibling module declaring its `CommandContext` const (with `Messaging.label`), its
private `as const` status tuple, its render map total over its own statuses (lifting
the corresponding `renderPluginRow` switch arms verbatim), and its command-private
reasons. Each orchestrator threads `notifyWithContext(ctx, pi, <CMD>_CONTEXT, rows)`
at its call sites and annotates row cardinality via the Single/Plural aliases. Because
`notifyWithContext` dispatches the per-row body through `context.render[status]` (wired
in 01-01), each command's catalog-uat run is a full end-to-end validation of that
command's own render map.

Purpose: Relocate the plugin half of the central `renderPluginRow` switch and the
plugin status sets out of the monolith into the owning vertical slices (MOD-01/03),
making a missing render arm a compile error at the command module (D-10) — with zero
rendered-byte change.

Output: 7 new `.messaging.ts` sibling modules + the corresponding orchestrator
call-site rewires. Implements MOD-01/MOD-02/MOD-03 and OUT-07 cardinality typing for
the plugin family. Output-neutral: catalog-uat + notify-v2 stay green.

Scope note (scope-sanity): this plan touches 14 files (7 orchestrators + 7 new
messaging modules), above the 10-file warning threshold, because the work is
mechanically repetitive verbatim arm-lifting across one command family. The blast
radius is kept bisectable by the per-command byte-gate discipline below: catalog-uat +
notify-v2 are run after EACH individual command migration, never deferred to task end.
</objective>
