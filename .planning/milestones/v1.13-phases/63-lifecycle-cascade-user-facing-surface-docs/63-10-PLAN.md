# Classifier asymmetry â€” exact shape

Two functions consume the same `resolver.notes` strings and MUST agree on the closed-set REASONS mapping for the same prefix family:

1. **Probe surface** (`shared/probe-classifiers.ts::narrowResolverNotes`, lines 87-123): used by `/claude:plugin info` and `/claude:plugin list`. Recognizes four `hooks.json` prefixes:
   - `note.startsWith("hooks.json is not valid JSON:")`
   - `note.startsWith("hooks.json failed schema validation:")`
   - `note.startsWith("unsupported hooks:")`
   - `note.startsWith("malformed hooks.json:")`
   Maps all four â†’ `"unsupported hooks"` REASONS token. Then `note.includes("lspServers")` â†’ `"lsp"`. Catch-all â†’ `"unsupported source"`.

2. **Install surface** (`orchestrators/plugin/install.ts::narrowResolverReasons`, lines 1689-1736): used by the install cascade. Recognizes:
   - `manifestFieldTokenFromNote(reason)` carve-out for `contains lspServers`
   - `reason.includes("source")` â†’ `"unsupported source"`
   - Errno-substring fallbacks for `EACCES`/`EPERM`/`ENOENT`/`ENOTDIR`/`SyntaxError`
   - Empty-fallback â†’ `"unsupported source"`

   **Missing:** any arm matching the four `hooks.json` prefixes. The note `"malformed hooks.json: hooks.json failed schema validation: /description: ..."` (the exact note the resolver wraps from `parseHooksConfig`) does NOT contain the substring `"source"` and is NOT a manifest-field carve-out, so it falls through to the empty-fallback `"unsupported source"`.

The fix mirrors the probe-side arm verbatim. Insert it as the FIRST arm in the `for (const reason of reasons)` loop (after the empty-skip continue), BEFORE the manifest-field carve-out. Reasoning: `startsWith` is more specific than `includes`; the manifest-field carve-out and the `source`-includes arm are both substring-bearing and would not match a `hooks.json:`-prefixed note anyway, but ordering the most-specific arm first is the safe pattern and matches the probe-side ordering.

# Cross-surface parity test (exact spec)

New file: `tests/orchestrators/plugin/cross-surface-reason-parity.test.ts`.

Goal: pin that `narrowResolverReasons` (install surface) and `narrowResolverNotes` (probe surface) emit the SAME closed-set REASONS token for the same input note. Today's failure modes both sides should agree on:

1. `"hooks.json is not valid JSON: Unexpected token..."`  â†’ both â†’ `"unsupported hooks"`
2. `"hooks.json failed schema validation: /description: ..."` â†’ both â†’ `"unsupported hooks"`
3. `"unsupported hooks: (a) regex matcher in PreToolUse: /foo.*/"` â†’ both â†’ `"unsupported hooks"`
4. `"malformed hooks.json: hooks.json failed schema validation: ..."` â†’ both â†’ `"unsupported hooks"`
5. `"contains lspServers"` â†’ probe â†’ `"lsp"`; install â†’ `"lsp"` (via the existing manifest-field carve-out â€” both surfaces already agree, pin it as a sanity row).
6. `"some other unsupported source"` â†’ both â†’ `"unsupported source"` (the catch-all fallback agrees on both surfaces â€” pin as sanity row).

Implementation approach: this test is a pure-function test against the two classifier functions, NOT an end-to-end orchestrator test. Both functions are pure and total; calling them directly with the same input note avoids the need for filesystem fixtures and orchestrator wiring. This is the simplest closed-form way to pin the parity contract structurally.

**Access pattern:**
- `narrowResolverNotes` is exported from `shared/probe-classifiers.ts` â€” import directly.
- `narrowResolverReasons` is currently a private function inside `install.ts` (no `export` keyword, no `__test_*` re-export). It needs a test seam. Add an export alongside the existing test seams at the bottom of `install.ts` (the file already has `export { classifyEntityShapeError as __test_classifyEntityShapeError }` at line 1753 and `export { classifyInstallFailure as __test_classifyInstallFailure }` at line 1754):

The `__test_*` prefix is the established naming convention in `install.ts` for test-only seams. Place the new export immediately after the existing two seams to preserve grouping.

**Test body skeleton (executor adapts for actual paths and conventions):**

```
import assert from "node:assert/strict";
import test from "node:test";

import { __test_narrowResolverReasons } from "../../../extensions/pi-claude-marketplace/orchestrators/plugin/install.ts";
import { narrowResolverNotes } from "../../../extensions/pi-claude-marketplace/shared/probe-classifiers.ts";

// Cross-surface parity (HOOK-03 / LIFE-01 / SURF-01): the install cascade
// classifier `narrowResolverReasons` and the read-only probe classifier
// `narrowResolverNotes` MUST emit the SAME closed-set REASONS token for the
// SAME resolver-emitted note. Without this contract, the same on-disk
// condition surfaces with different (unavailable) {<reason>} tokens
// depending on which command the user runs (info/list vs install) â€”
// violating the same-plugin-same-reason invariant.
const PARITY_CASES = [
  { note: "hooks.json is not valid JSON: Unexpected token ] in JSON at position 5", expected: "unsupported hooks" },
  { note: "hooks.json failed schema validation: /description: expected array", expected: "unsupported hooks" },
  { note: "unsupported hooks: (a) regex matcher in PreToolUse: /foo.*/", expected: "unsupported hooks" },
  { note: "malformed hooks.json: hooks.json failed schema validation: /description: expected array", expected: "unsupported hooks" },
  { note: "contains lspServers", expected: "lsp" },
  { note: "some other unsupported source detail", expected: "unsupported source" },
] as const;

for (const { note, expected } of PARITY_CASES) {
  test(`cross-surface parity: "${note.slice(0, 40)}..." -> "${expected}" on both surfaces`, () => {
    const probeOut = narrowResolverNotes([note]);
    const installOut = __test_narrowResolverReasons([note]);
    assert.deepEqual(probeOut, [expected], `probe surface emitted ${JSON.stringify(probeOut)}`);
    assert.deepEqual(installOut, [expected], `install surface emitted ${JSON.stringify(installOut)}`);
  });
}
```

The test docstring + the per-case `assert` messages capture the parity contract. The test is purely closed-form (no fs, no spawn, no async, no fixtures).
