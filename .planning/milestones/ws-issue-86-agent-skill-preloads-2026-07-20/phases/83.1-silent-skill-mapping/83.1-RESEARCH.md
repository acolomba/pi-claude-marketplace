## Summary

This is a surgical, two-source-file + three-test-file change. The phase-82 and
phase-83 Skill warning strings live in a single ternary inside `mapTools`
(`convert.ts:278-284`) -- deleting that block removes both strings in one
edit. `Skill` reaches `droppedTools` through the generic TOOL_MAP-miss path in
`mapToolTokens` (`convert.ts:230-237`); excluding it there (recommended seam)
silences every downstream consumer at once: the provenance HTML comment, the
`ConvertedAgent` record, the persisted `agents-index.json` entry, and the
install-time `dropped tools:` notify line in `stage.ts` -- none of which need
their own edits. The legend's two-vs-three-state selection is a single ternary
in `renderSkillLegend` (`frontmatter.ts:342-344`); collapsing it to the
unconditional `"available on demand"` string and dropping the now-unused
`inheritSkills` parameter (private function, single call site at
`frontmatter.ts:318`) completes the source changes.

The test blast radius is fully enumerated below: 2 warning-string constants
and ~10 tests in `convert.test.ts` (including two whole-file pins that must be
deliberately re-pinned per D-83.1-04), 1 expected-constant line in
`frontmatter.test.ts`, and ZERO edits to `convert-byte-identity.test.ts` --
its seven corpus classes contain no Skill declarations, no body tokens, and no
legends, so they are the phase's built-in SC-4 guard. No test outside these
three files references `inheritSkills`, the Skill tool, or any of the strings
being changed (verified by repo-wide grep).

**Primary recommendation:** Exclude `Skill` inside `mapToolTokens` (skip:
neither mapped nor dropped), delete the warning block in `mapTools`, collapse
the legend ternary in `renderSkillLegend` and remove its `inheritSkills`
parameter, then update exactly the pinned tests enumerated in the Current
State Map -- and verify `convert-byte-identity.test.ts` has a zero-line diff.

## Common Pitfalls

### Pitfall 1: Filtering droppedTools at provenance assembly only
**What goes wrong:** `convertAgent` threads `toolsResult.dropped` to TWO
places (`convert.ts:550` provenance, `convert.ts:564` ConvertedAgent record).
Filtering only one makes the HTML comment disagree with `agents-index.json`
and the install notify.
**How to avoid:** Seam A (mapToolTokens) -- single upstream exclusion.
**Warning sign:** stage.test/index tests would not catch it (they use empty
fixtures); only a targeted droppedTools deepEqual on `convertAgent`'s RESULT
(not just fileContent) does. Keep both kinds of assertions in the rewritten
tests (the current tests already assert on `out.droppedTools`, the result --
preserve that).

### Pitfall 2: Editing the byte-identity corpus to make something pass
**What goes wrong:** D-83.1-04 explicitly requires the seven classes stay
untouched; any red there means the source change leaked beyond the Skill/
legend surface.
**How to avoid:** Treat `convert-byte-identity.test.ts` as read-only; add the
`git diff --stat` check to the plan's verification step.

### Pitfall 3: Leaving the unused `inheritSkills` parameter on renderSkillLegend
**What goes wrong:** After collapsing the ternary the parameter is dead;
typescript-eslint flags unused parameters and `npm run check` goes red.
**How to avoid:** Remove the parameter AND the second argument at
`frontmatter.ts:318` in the same edit.

### Pitfall 4: Deleting `skillDeclared` along with the warning block
**What goes wrong:** `skillDeclared` (convert.ts:269) is used by BOTH the
deleted warning `if` (L278) and the kept `inheritSkills` computation (L270).
Overzealous cleanup breaks the fenced-off behavior.
**How to avoid:** Delete exactly L274-284; typecheck catches it if not.

### Pitfall 5: Old "never edit this constant" guard comments taken literally
**What goes wrong:** `DISALLOWED_SKILL_EXPECTED` (convert.test.ts:632-641)
says "Never edit this constant to make a converter change pass" -- but it was
pinned to the now-superseded pre-AGSK-05 contract. An executor following the
comment verbatim will refuse the required re-pin.
**How to avoid:** The plan task must explicitly state this is the D-83.1-04
"deliberate update of affected pinned fixtures" and must rewrite the guard
comment to describe the new (amended AGSK-03) contract it pins.

### Pitfall 6: grep sweep silently skipping glyph files on macOS
**What goes wrong:** BSD grep treats files containing multi-byte glyphs as
binary and skips them without matching (known repo issue with U+2192-bearing
files).
**How to avoid:** Always `grep -a` (or a Node scan) for the verification
sweeps. The affected .ts files here use `→` escapes (ASCII source), but
`-a` costs nothing and removes the hazard class.

### Pitfall 7: Comment-policy violations in new comments/titles
**What goes wrong:** Writing `// Phase 83.1: ...` or test titles citing
superseded decision IDs; pre-commit/review flags them, or worse, they land
and rot.
**How to avoid:** Use only AGSK-NN / D-83.1-NN / #86 anchors per item 7.

### Pitfall 8: Warnings-order test left asserting a removed warning
**What goes wrong:** `convert.test.ts:676-689` deep-equals the full warnings
array including `SKILL_INHERIT_WARNING`; deleting the constant without
rewriting the expectation is a compile error (good), but rewriting it to
"just delete the test" loses the valuable inverse pin.
**How to avoid:** Keep the test, expected array becomes exactly
`[crossPluginSkillWarning("other-plugin:x")]` -- it now proves Skill
contributes nothing even when other warnings fire.

### Seam A exclusion in mapToolTokens (convert.ts)

```typescript
// AGSK-03 / D-83.1-02: Skill is translated to inheritSkills (computed in
// mapTools from raw tokens), never dropped -- it produces no droppedTools
// entry and no warning in any branch. Exact match, like TOOL_MAP lookups.
function mapToolTokens(tokens: readonly string[]): { mapped: string[]; dropped: string[] } {
  const mapped: string[] = [];
  const dropped: string[] = [];
  for (const token of tokens) {
    if (token === "Skill") {
      continue;
    }
    const piName = TOOL_MAP[token];
    if (piName === undefined) {
      dropped.push(token);
    } else {
      mapped.push(piName);
    }
  }
  return { mapped, dropped };
}
```
