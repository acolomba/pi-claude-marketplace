<decisions>
## Implementation Decisions

### Silent Skill conversion (amended AGSK-03/05)
- **D-83.1-01:** No Skill-specific warning in ANY branch. Both existing
  strings are removed: the phase-82 string ("generated agents run with
  skills discovery disabled...") and the phase-83 string ("mapped to Pi
  skill discovery..."). Rationale (user decision): declared-and-allowed →
  `inheritSkills: true` reproduces Claude's Skill-tool capability;
  declared-and-disallowed → suppressed, and the user verified Claude Code
  itself lets disallowedTools win. Both branches operate exactly as Claude
  Code does — there is nothing to warn about.
- **D-83.1-02:** `Skill` never appears in `droppedTools` — not when mapped
  to `inheritSkills: true` (it was translated, not dropped) and not in the
  disallowed branch (Claude-consistent silent suppression). Genuinely
  unmapped tools (e.g. `WebFetch`) keep their `droppedTools` entries
  unchanged.

### Unified legend state (amended AGSK-04)
- **D-83.1-03:** The legend's non-preloaded annotation is
  "(available on demand)" for ALL agents regardless of `inheritSkills`.
  "(not available in this session)" is removed from the codebase.
  Empirical basis (UAT 2026-07-19): pi's `--no-skills` keeps explicit
  `--skill` flags and extension-contributed paths
  (`pi-coding-agent/dist/core/resource-loader.js:277`); a `--no-skills`
  child in a real environment listed every marketplace-converted skill,
  re-injected via the pi-claude-marketplace extension's
  `resources_discover`. Converted skills are catalog-present in child
  sessions either way, so the state distinction was false.

### Byte-identity contract
- **D-83.1-04:** Relative to Phase 83 output: agents WITHOUT a `Skill`
  declaration and WITHOUT body skill tokens stay byte-identical. Agents
  with `Skill` declarations change only in provenance (warning gone,
  droppedTools entry gone). Agents with body tokens referencing
  non-preloaded skills change only the legend state text. Update the
  affected pinned fixtures deliberately; the seven-class byte-identity
  corpus classes not involving Skill or legend states must stay untouched.

### Claude's Discretion
- Mechanics of excluding Skill from droppedTools (filter at mapTools vs
  at provenance assembly) — pick the seam that keeps the logic in one
  place and test-pin the outcome.

</decisions>
