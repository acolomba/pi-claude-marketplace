## Summary

Reviewed the silent-Skill-conversion change (commit 0e634e6f) and the unified
legend annotation change (commit 3f1e78d4) in `convert.ts` / `frontmatter.ts`
plus their test suites. Both test suites pass (91/91). The three requested
scrutiny points were verified in depth:

1. **`droppedTools` downstream consumers behave sensibly.**
   `stage.ts:244` copies `droppedTools` verbatim into the agents-index entry;
   `stage.ts:299-301` (`formatAgentWarnings`) only emits the
   `dropped tools:` notify line when the array is non-empty. Neither has any
   Skill-specific branch, so the silence propagates cleanly from the single
   upstream seam. `persistence/agents-index-schema.ts:36` types the field as
   `Type.Array(Type.String())` with no enum or minItems, so previously
   persisted index files that still contain `"Skill"` in `droppedTools`
   continue to validate and are naturally rewritten on the next
   install/update. No other consumer of `droppedTools` exists
   (repo-wide grep).

2. **The `mapToolTokens` Skill skip cannot mask a genuinely unmapped tool.**
   The skip (`convert.ts:233-235`) is an exact-match `token === "Skill"`
   `continue` inside a per-token loop; every other `TOOL_MAP` miss still
   lands in `dropped` (`convert.ts:237-243`). Case variants (`skill`) and
   near-matches (`SkillFoo`) fall through to the dropped path, consistent
   with the exact-match `TOOL_MAP` lookup policy. One coverage gap on this
   exact property is flagged as WR-02.

3. **No dead code from the removed warning strings.** Both deleted Skill
   warning strings, the retired `(not available in this session)` annotation,
   and the dead `inheritSkills` parameter of `renderSkillLegend` are fully
   gone. No orphaned constants, no unused imports, no leftover references in
   `docs/output-catalog.md` (checked with `grep -a` to defeat BSD grep's
   binary-glyph skip) or the PRD. `skillDeclared` (`convert.ts:275`) remains
   live -- it feeds the `inheritSkills` computation. The byte-identity corpus
   contains no Skill-declaring fixture, consistent with its zero-diff claim.

Additional traced properties: `detectSkillTokens` is throw-safe -- its
candidate class `[A-Za-z0-9_-]+` excludes every `assertSafeName` rejection
(empty, `.`/`..`, path separators, control chars) except the
empty-after-elision case, which the `candidate === `${pluginName}-`` guard at
`convert.ts:165` covers (verified against `domain/name.ts:64-77`). The
`inheritSkills` flag is computed once from raw tokens before mapping, so the
Skill skip cannot affect it, and the `disallowedTools`-only case correctly
never flips it.

## Narrative Findings (AI reviewer)

### Warnings

#### WR-01: AG-11 failure for a Skill-only agent is now fully opaque

**File:** `extensions/pi-claude-marketplace/bridges/agents/convert.ts:482-491`
**Issue:** A legal Claude Code agent declaring `tools: Skill` (skill
invocation only, no other tools) now converts to an empty mapped tool list
and hits the AG-11 throw, failing the entire agents staging step
(`stage.ts:134-144` has no per-agent catch). The throw itself predates this
phase, but 83.1 deleted the only user-visible string that explained the
Skill-to-`inheritSkills` translation. The resulting error --
`the mapped tool list is empty ... Source tools: Skill; disallowedTools: (none).`
-- gives the user no clue why the one tool they declared produced zero mapped
tools, since Skill is now silently excluded from classification everywhere
(no `droppedTools` entry, no warning, no provenance line). There is also no
test covering the Skill-only input.
**Fix:** Append a Skill note to the AG-11 message when Skill was among the
raw tokens, and pin the Skill-only case in a test:
```ts
if (toolsResult.mapped.length === 0) {
  const skillNote = splitCsv(raw.tools).includes("Skill")
    ? " Note: the Skill tool maps to inheritSkills, not to a Pi tool, so it does not count toward the tool list."
    : "";
  throw new Error(
    `Cannot convert agent "${sourceName}" in plugin "${pluginName}": ` +
      `the mapped tool list is empty (pi-subagents has no safe representation of "no tools"). ` +
      `Source tools: ${raw.tools ?? "(default read,bash,edit)"}; ` +
      `disallowedTools: ${raw.disallowedTools ?? "(none)"}.${skillNote}`,
  );
}
```

#### WR-02: No regression test for Skill co-occurring with a genuinely unmapped tool

**File:** `tests/bridges/agents/convert.test.ts:585-596`
**Issue:** The silent-skip contract is pinned only in isolation: `Bash, Read,
Skill` asserts empty `droppedTools` (line 585), and `Read,WebFetch` asserts
`["WebFetch"]` without Skill present (line 592). No test asserts the combined
case, which is exactly the regression class this phase creates: an
implementation that suppresses `dropped` entirely whenever Skill is declared
(a plausible misreading of "no droppedTools entry ... in any branch") would
pass every existing test while silently swallowing real unmapped tools.
**Fix:** Add the combined pin:
```ts
test("AGSK-03 Skill alongside an unknown tool still records the unknown drop", () => {
  const out = convertSpecTree({ description: "d", tools: "Read, Skill, WebFetch" });
  assert.deepEqual([...out.droppedTools], ["WebFetch"]);
  assert.deepEqual([...out.warnings], []);
});
```

#### IN-02: Whole-file pins mix an `acme` generated name with a `spec-tree` plugin

**File:** `tests/bridges/agents/convert.test.ts:628-648,947-968`
**Issue:** `DISALLOWED_SKILL_EXPECTED` and `preLegendExpected` pin
`name: pi-claude-marketplace-acme-bot` alongside `plugin: spec-tree` because
`convertSpecTree` reuses `makeDiscovered`'s `acme` default `generatedName`
while passing `pluginName: "spec-tree"`. The pins accurately capture
`convertAgent`'s pass-through of `discovered.generatedName`, so nothing fails
today, but the fixture files are internally inconsistent and could mislead a
maintainer reading them as realistic examples (a real `spec-tree` agent would
be named `pi-claude-marketplace-spec-tree-bot`).
**Fix:** Pass `generatedName: "pi-claude-marketplace-spec-tree-bot"` through
`makeDiscovered` in `convertSpecTree` and deliberately re-pin the two
constants (an intentional fixture correction, not a converter change).
