Re-attribute the install and uninstall marketplace-existence / scope preconditions to the
canonical `info` model, and land the cross-scope discriminated resolver foundation in the shared
chokepoint that this plan (uninstall) and Plan 47-03 (reinstall/update) both consume.

marketplace-subject output `info` already emits (Phase 46 `MarketplaceNotAddedMessage`), instead of

`{not in manifest}` cleanup-cascade reason (M5). "Plugin absent from a present manifest" stays
`{not in manifest}` (install M2 -- the heart of ATTR-08).

`uninstall.ts` (M3/M4 loud + M4b silent + ATTR-09 reasons), with paired catalog states + fixtures.

<resolved_open_questions>
This plan records the phase's three open-question resolutions (RESEARCH §Open Questions):

- **Cascade emission model (D-47-A):** STANDALONE top-level `MarketplaceNotAddedMessage` emission,
  matching `info` exactly. The marketplace-absent / wrong-scope precondition is checked BEFORE any
  cascade row exists, and the variant is a top-level `NotificationMessage` arm that is NOT a
  `PluginNotificationMessage` (embedding as a cascade row is not type-representable). Each miss does
  `notify(ctx, pi, { kind: "marketplace-not-added", name, scope? }); return;`.
- **SCOPE-01 byte surface (Open Question #1 / Assumption A2):** RESOLVED to the
  `[requestedScope]`-bracketed `{not added}` form -- byte-identical to `info`'s
  `missing-marketplace-not-added-scope-mismatch` state (`docs/output-catalog.md:1024-1032`). No new
  renderer affordance, no new REASONS member, no type-model touch. The `[scope]` bracket
  communicates "not added in the scope you asked for"; the operator infers the other scope. A richer
  "present in <other> scope" phrase would expand the byte surface and touch the renderer -- explicitly
  REJECTED for the milestone's lean (locked in 47-CONTEXT Out-of-Scope: no new member, info is the
  model). Recorded so Plan 47-02/03 stay consistent.
- **Reason vocabulary (D-47-B):** reuse existing `REASONS` members; NO new member. See Task 2.
</resolved_open_questions>
