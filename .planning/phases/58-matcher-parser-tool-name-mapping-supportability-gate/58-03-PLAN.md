<objective>
Extend `domain/components/hooks.ts` with the matcher parser
(`parseMatcher`), the supportability gate (`checkMatcherSupportability`),
and the single-seam extension of `parseHooksConfig` (D-58-03) so a
plugin whose `hooks.json` passes Phase 57's JSON.parse + schema check
but trips ANY of TOOL-02(a)/(b)/(c)/(d) flips
`installable: false` with reason `"unsupported hooks: ..."`. Per-condition
detail goes to `hookDebugLog` only; the user-facing reason is the
single literal `"unsupported hooks"` after Plan 04's REASONS rename.

In this plan the reason string is the prefixed PARSE-TIME form
`"unsupported hooks: ..."`. Plan 04's `narrowResolverNotes`
tightening will read this prefix and emit the closed-set `"unsupported hooks"`
Reason member.

Purpose: Lock the MATCH-01 / MATCH-02 / TOOL-02 contracts in one
leaf-pure parser module. Phase 59's dispatcher (later) will read the
`{kind:"tool-set", piTools:Set<PiToolName>}` and
`{kind:"match-all"}` / `{kind:"mcp-literal", literal}` arms; the
`{kind:"regex"}` / `{kind:"unmapped"}` arms never reach the
dispatcher because the plugin flips unavailable at parse time.

Output: Extended `hooks.ts` (~180 → ~450 lines) + populated unit
test file + appended architecture-test invariants on
`checkMatcherSupportability`.

## Artifacts this phase produces

- `ParsedMatcher` — discriminated union with 5 arms:
  * `{ kind: "match-all" }` (for `""` or `"*"`).
  * `{ kind: "tool-set"; piTools: ReadonlySet<PiToolName> }` (for one
    Claude tool name or pipe-OR alternation of Claude tool names).
  * `{ kind: "mcp-literal"; literal: string }` (for `mcp__server__tool`).
  * `{ kind: "regex" }` (for MATCH-02 regex chars or malformed pipe-OR).
  * `{ kind: "unmapped"; token: string }` (for Claude-form tokens with no
    TOOL-01 mapping — `MultiEdit` / Pi-form lowercase / etc.).
- `parseMatcher(raw: string): ParsedMatcher` — pure, never throws.
- `checkMatcherSupportability(config: HooksConfig): { ok: true } | { ok: false; debugDetail: string }`
  — pure, never throws; the FIRST failing entry determines the
  debugDetail.
- Extended `parseHooksConfig(raw: string): HookConfigParseResult` —
  after the existing JSON.parse + HOOKS_VALIDATOR.Check arms, calls
  `checkMatcherSupportability` and folds failures into the existing
  `{ ok: false, reason }` arm with reason
  `"unsupported hooks: " + debugDetail` (D-58-03 single seam).
- `hookDebugLog` is called with each PARSE-TIME-resolved reason (including
  the supportability failure detail) — already existing pattern from
  Phase 57 baseline.
</objective>
