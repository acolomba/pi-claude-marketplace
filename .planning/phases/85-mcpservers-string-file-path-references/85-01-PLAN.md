<objective>
Teach the strict resolver to accept a `mcpServers` declared as a `./`-relative
string path (from a `marketplace.json` plugin entry OR a `plugin.json`) that
points at a WRAPPED `.mcp.json` inside the plugin root, and install the declared
servers at full parity with the inline-object form. A missing / malformed /
wrapper-less / out-of-root reference degrades that ONE plugin to `(unavailable)`
with a note, never failing the whole marketplace load.

Purpose: Deliver MCPR-01..04 in the single resolver seam the roadmap chose
(`domain/resolver.ts::applyStrictMcp`, before `applyMcpValue`), reusing
`assertPathInside` for containment (NFR-10 + D-14). The schema widening is a
hard PREREQUISITE: object-only `mcpServers` schemas make a string entry throw the
whole-manifest validation error the requirements forbid.

Output: A widened `mcpServers` field union in both plugin schemas; two new local
resolver helpers (`validateReferencePath`, `readReferencedMcp`) plus a string
branch in `applyStrictMcp`; and the full resolver + schema-acceptance test matrix.
This plan does NOT wire the `{malformed mcp}` reason token -- that is Plan 02.
Error references here surface a `malformed mcp reference:` note (which narrows to
`{unsupported source}` until Plan 02 lands the token); resolver tests assert note
CONTENT + `state`, not the narrowed token, so ordering is safe.
</objective>
