### Gaps Summary

**Goal achievement is BLOCKED by CR-01, an unresolved critical finding from the phase's own code review.** The symlink-escape walker in `bridges/hooks/stage.ts:55-107` correctly raises `SymlinkRefusedError` when a symlinked-escape is present, so the test suite passes (`tests/bridges/hooks/symlink-escape.test.ts` Cases A+B). However, the underlying enumerator at line 59 calls `readdir(hooksRoot, { recursive: true, withFileTypes: true })`, and Node's recursive readdir FOLLOWS symbolic links to directories. The consequences (per the 63-REVIEW.md analysis):

1. **External-tree FS reads occur before rejection.** A malicious plugin can probe arbitrary attacker-chosen paths (`~/.ssh`, `/proc/self`, etc.) by symlinking them in; the enumerator reads the contents BEFORE the symlink entry is processed and rejected.
2. **The user-visible error message names an external path** as the "hooks subtree symlink" — `entry.parentPath` for entries discovered inside the followed symlink points outside `pluginRoot`.
3. **Non-symlink entries inside the followed target are silently skipped**, so a nested symlink loop inside the external dir is not examined defensively.

The LIFE-03 success-criterion wording ("a symlinked-escape command path is rejected at install with a notify error via fs.realpath + assertPathInside(<pluginRoot>, realpath)") is technically satisfied for the eventual rejection, but the criterion's INTENT (refuse to read outside pluginRoot) is violated. The threat model in `63-02-PLAN.md::T-63-LIFE-03-SE` explicitly says "first escape throws `SymlinkRefusedError`" but does not address the pre-rejection reads. The plan's `<verification>` block (`grep -c "assertPathInside" stage.ts >= 2`) was met but does not pin the walker's containment.
