# Wire-format authority

The upstream Claude Code `plugin-dev/skills/hook-development/SKILL.md` is the binding contract for the plugin `hooks/hooks.json` shape. Verbatim quote (recorded in the debug session at `.planning/debug/hookify-unavailable-resolver-flip.md`):

> For plugin hooks in `hooks/hooks.json`, use **wrapper format**:
>   `{ 'description': 'Brief explanation of hooks (optional)', 'hooks': { 'PreToolUse': [...], ... } }`
> Key points:
> - description field is optional
> - hooks field is REQUIRED wrapper containing actual hook events
> - This is the **plugin-specific format**
>
> For user settings in `.claude/settings.json`, use **direct format**:
>   `{ 'PreToolUse': [...], 'Stop': [...], ... }`
> Key points:
> - No wrapper - events directly at top level
> - No description field
> - This is the **settings format**

The current `HOOKS_CONFIG_SCHEMA` encodes the SETTINGS format but is applied to plugin files. The fix unwraps the wrapper when present, preserving validation of the bare shape for any in-tree code paths that may legitimately produce it.

# Wrapper-detection heuristic (exact spec)

Inside `parseHooksConfig`, AFTER `JSON.parse` succeeds and BEFORE `HOOKS_VALIDATOR.Check`, insert:

```
1. If parsed is a plain non-null non-array object AND
2. parsed has an own enumerable `hooks` property AND
3. parsed.hooks is a plain non-null non-array object
THEN candidate = parsed.hooks
ELSE candidate = parsed
```

Then validate `candidate` against `HOOKS_VALIDATOR.Check`. The downstream `value` returned in the `{ok: true, value, ifPredicates}` arm is `candidate` (the unwrapped record), NOT the outer wrapper. This keeps every downstream consumer (resolver, info.ts projection, bridge write path) unchanged — they continue to consume the bare-event-keys record they already expect.

Implementation note: use `typeof v === "object" && v !== null && !Array.isArray(v)` for the plain-object check. Do NOT trust `instanceof Object` (cross-realm hazard, irrelevant here but bad practice). The `hooks` property check uses `Object.prototype.hasOwnProperty.call(parsed, "hooks")` to avoid prototype-chain false positives.

Edge cases this heuristic correctly handles:
- `{description: "x", hooks: {PreToolUse: [...]}}` → unwrap → validate `parsed.hooks`. (Hookify case.)
- `{PreToolUse: [...]}` (bare top-level event keys, no `hooks` property) → no unwrap → validate `parsed` (backward-compatible — existing WR-05-style fixtures still work even if NOT reverted by this plan; we revert them anyway for correctness).
- `{hooks: [...]}` (parsed.hooks is an array, not an object) → no unwrap → validate `parsed`. The validation will fail on `hooks: [...]` because the schema expects an array per event key, not a top-level array; this rejection is correct (the input is malformed under both formats).
- `{hooks: {}}` (empty wrapper) → unwrap to `{}` → `HOOKS_VALIDATOR` accepts the empty record (it's a `Type.Record(Type.String(), ...)`, which permits zero entries). Same behavior as today's behavior on `{}`.
- `null` / `[]` / a string / a number after `JSON.parse` → neither the plain-object check nor the `hooks` property check succeeds → validate the raw value → `HOOKS_VALIDATOR.Check(null)` returns false (root type mismatch) → the existing `{ok: false, reason: "hooks.json failed schema validation: ..."}` arm fires. No behavior regression.
