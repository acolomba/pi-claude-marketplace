<objective>
Close RESEARCH adjudication item 2 (Phase 48 IN-02): for the SAME on-disk condition -- a schema-invalid
`marketplace.json` -- the WRITE path (`marketplace add` via `classifyAddError`) renders `{invalid
manifest}` while the READ path (`marketplace info` / `plugin info` / `list` via `narrowProbeError`)
falls through to the permissive `{unreadable}` fallback. That cross-surface asymmetry is exactly the
precondition-class-3 (manifest-failure) inconsistency this phase closes. The user-facing REASON
describes the manifest's state (it is invalid), which is identical regardless of read vs write -- the
asymmetry is a leak of the classifier's internal ladder, not a real semantic distinction.

Purpose: one-line-arm parity fix. `invalid manifest` is ALREADY a REASONS member (notify.ts), so this
is a classifier widening, not a new member. Byte-additive: the malformed-JSON `{unparseable}` arm is
preserved (collapsed into the single InvalidMarketplaceManifestError branch); the generic
`{unreadable}` fallback for non-manifest errors (EIO, unknown shapes) is untouched.

Output: the widened `narrowProbeError` ladder, its widened return-type union threaded through
`ListReason` / the `marketplace info` reason field, unit cases via `__test_narrowProbeError` in both
read-surface test files, and one new `marketplace info` manifest-invalid catalog state + paired
fixture so the read-surface `{invalid manifest}` byte form is byte-locked.
</objective>
