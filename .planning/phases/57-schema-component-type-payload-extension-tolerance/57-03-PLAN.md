<objective>
Move the `hooks` component out of the resolver's unsupported set and into the supported set, so a plugin declaring `hooks` parses through the discriminated `installable: true | false` resolver and lands as `installable: true` when its `hooks/hooks.json` is valid (or `installable: false` with parse-failure detail when not). Add `hooksDir` to `ScopedLocations` so Phase 63 LIFE-03 has a hard-coded containment target to assert against.

Purpose: Per HOOK-01, the resolver must recognize `hooks` alongside `skills` / `commands` / `agents` / `mcpServers`; per D-57-04, parse-failure routes to `installable: false`. This plan is the resolver-side admission move + the persistence-layer location hook for the next milestone phase. The actual bridge dispatch, exec, surface, and notification work all live in Phases 58–63; Phase 57's job is to make the resolver observable.

Output: Updated `SUPPORTED_COMPONENT_KINDS`, removed `hooks` from `UNSUPPORTED_COMPONENT_KINDS` / `UNSUPPORTED_COMPONENT_CONVENTIONS`, new resolver branch reading the hooks file and routing through `parseHooksConfig`, new `hooksDir` field on `ScopedLocations`, and behavior tests on both the strict and loose resolver paths plus the locations bundle.
</objective>
