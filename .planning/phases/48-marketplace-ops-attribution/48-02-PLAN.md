<objective>
Converge the missing-marketplace precondition for `marketplace autoupdate` / `noautoupdate`
(ATTR-05) and `marketplace remove` (ATTR-06) on the Phase 46 standalone `MarketplaceNotAddedMessage`
`(failed) {not added}` variant -- the canonical `info` model. This reuses Phase 46/47 primitives
verbatim (no new mechanism): the `MarketplaceNotAddedMessage` variant + `renderMarketplaceNotAdded`,
and (where the signal-catch shape fits) the exported `MarketplaceNotAddedSignal` from
`orchestrators/plugin/shared.ts`.

Today:
- `autoupdate` S1 (autoupdate.ts:167-187 + `autoupdateFailedRow` :129-138): an explicit-scope
  `MarketplaceNotFoundError` renders `(failed)` with a synthetic child `{not found}` -- wrong reason.
- `autoupdate` S2 (autoupdate.ts:196-215): a name missing in EVERY scope renders a reason-LESS bare
  `(failed)` -- no reason at all.
- `remove` S3 (remove.ts:182-185): a missing record throws `MarketplaceNotFoundError` raw INSIDE the
  guard → escapes `removeMarketplace` (no catch) → the edge handler (no catch) → raw to the Pi runner.
- `remove` S4 (remove.ts:167-169): the bare form's `resolveScopeFromState` throws
  `MarketplaceNotFoundError` raw when the name is in neither scope -- same raw escape.

Per Decision D-48-C Shape 1 (the Phase 47 convention -- "the fix is at the orchestrator, NOT the
edge"), all four route through `notify` INSIDE the orchestrator. The `StateLockHeldError →
(failed) {lock held}` autoupdate path is NOT a missing-marketplace and stays unchanged.

Purpose: Make autoupdate/noautoupdate/remove of a missing marketplace consistent, truthful, and
non-raw-throwing -- matching the canonical `{not added}` model.
Output: Re-routed autoupdate S1/S2 + remove S3/S4; new catalog states + fixtures; amended tests.
</objective>
