# Phase 14.1: Code Review Report

**Reviewed:** 2026-05-24
**Depth:** standard
**Files Reviewed:** 5
**Status:** issues_found (3 Info, no Critical or Warning)

## Summary

Phase 14.1 closes CMC-13 by promoting `declaresAgents` / `declaresMcp` from a
silently-defaulted `false` at the cascade-row build site to REQUIRED fields on
both `InstallPluginOutcome.installed` and `PluginInstalledOutcome`, with
plumbing through `import/execute.ts`'s `case "installed"` mapping.

The typed-plumbing change is correct and matches the standalone success-row
expression form (`installCtx.stagedAgentNames.length > 0` /
`installCtx.stagedMcpServerNames.length > 0`). The 4 new propagation tests
exercise all four predicate combinations (true/false x agents/mcp) and assert
both the raw outcome shape and the rendered cascade body via `UNLOADED_PROBE`.
The new `soft-dep-markers` catalog fixture matches the catalog text by
construction (single marketplace block, no `<autoupdate>` marker, joins with
`\n` matching the catalog's zero-blank-line shape, paired through
`appendReloadHint` with the canonical reload hint).

No correctness, security, or maintainability defects identified. Three Info
items follow that surface comment-to-type-system drift and a small symmetry
gap with the failure-path predicates -- none block ship.

## Info

### IN-01: NFR-7 comment overstates the type-system guarantee

**File:** `extensions/pi-claude-marketplace/orchestrators/plugin/install.ts:120-121`
**Issue:** The new JSDoc on `InstallPluginOutcome` claims:

> Making them REQUIRED rather than optional honors NFR-7: a discriminated outcome must not have a third undefined state at consumers.

The REQUIRED-ness is enforced only at the `InstallPluginOutcome` boundary and
at `PluginInstalledOutcome` (the import-result shape). The downstream consumer
that actually consumes these predicates -- `PluginCascadeRow` in
`presentation/compact-line.ts:146-147` and `PluginInlineRow:104-105` /
`PluginListRow:177-178` -- still declares both fields as OPTIONAL
(`declaresAgents?: boolean`). The "third undefined state at consumers"
that NFR-7 forbids is therefore still reachable structurally at the renderer
boundary; only the orchestrator-to-orchestrator boundary is hardened. This
is functionally safe (`composeReasons` at `compact-line.ts:451`/`455` checks
`=== true` so `undefined` is correctly treated as `false`), but the comment
overstates what the change accomplishes.

**Fix:** Either tighten the comment to scope the claim accurately ("REQUIRED
at the install -> import boundary so consumers cannot silently default to
false"), or follow up by promoting `declaresAgents`/`declaresMcp` to required
on `PluginCascadeRow` (the only rendered-row shape touched by this PR's
emission site). The comment-only fix is sufficient for v1.3.

### IN-02: Failure-path predicates not propagated to orchestrated outcome

**File:** `extensions/pi-claude-marketplace/orchestrators/plugin/install.ts:600-601,656,954-972`
**Issue:** The success return at `install.ts:783-789` propagates
`declaresAgents`/`declaresMcp` onto the `"installed"` outcome variant. The
failure path captures the same predicates as `failureDeclaresAgents` /
`failureDeclaresMcp` (lines 600-601) and feeds them into
`composeRollbackPartialBody` for the standalone notification path (lines
638-647), but `classifyInstallFailure` (lines 954-972) -- the function that
builds the outcome returned to orchestrated callers like `import/execute.ts`
-- does NOT carry these predicates onto the `unavailable` / `uninstallable` /
`unexpected-failure` outcome variants. Today the import orchestrator only
reads predicates on the `"installed"` arm of the switch (execute.ts:846-867),
so this is not a live defect, but the asymmetry means a future feature that
wants soft-dep markers on cascade `(failed)` rows in import surfaces will
silently render as `false` (the same class of defect CMC-13 just closed for
`(installed)` rows).

**Fix:** Either (a) add `declaresAgents` / `declaresMcp` to the
non-`installed` outcome variants for parity with the success arm, OR (b) add
a comment at `classifyInstallFailure` noting that import-surface failed rows
DO NOT currently propagate the predicates and that adding them requires
widening the discriminated outcome. Option (b) is the v1.3 minimum.

### IN-03: Soft-dep-markers fixture parity bounded to single-block blocks

**File:** `tests/architecture/catalog-uat.test.ts:1140` and `docs/output-catalog.md:598-614`
**Issue:** The new `soft-dep-markers` fixture joins block bodies with `\n`:

```ts
return appendReloadHint(`Claude plugin import summary\n\n${bodies.join("\n")}`, RELOAD_HINT);
```

This matches the catalog text byte-for-byte for the new single-block fixture.
However, the pre-existing `fresh-mixed-both-scopes` / `scope-project-narrow`
/ `source-mismatch` fixtures use the same `bodies.join("\n")` pattern while
production code in `composeImportSummary` (`execute.ts:415`,
`spliceSourceMismatchDiagnostics` line 629 / 655 / 676) joins segments with
`\n\n` (blank line between marketplace blocks). The catalog and the fixture
agree (no blank line), but the production output diverges from both for
multi-block cases. The new soft-dep-markers fixture has only ONE marketplace
block, so it does not exercise the join discrepancy.

This is a PRE-EXISTING discrepancy, NOT introduced by phase 14.1, and is out
of strict scope. Surfacing it here because the soft-dep-markers fixture
inherits the existing pattern and would be the natural place to add a
multi-block soft-dep case in a follow-up.

**Fix:** Out of scope for 14.1. Track as a separate follow-up: either align
`composeImportSummary` segment-join to `\n` (matching the catalog), or update
the catalog and all multi-block fixtures to use the `\n\n` blank-line shape
that production emits. Pick one and make catalog UAT enforce it.
