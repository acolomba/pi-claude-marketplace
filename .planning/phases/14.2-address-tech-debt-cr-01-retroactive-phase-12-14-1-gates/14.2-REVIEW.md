# Phase 14.2: Code Review Report

**Reviewed:** 2026-05-24T22:00:00Z
**Depth:** standard
**Files Reviewed:** 14
**Status:** issues_found

## Summary

Phase 14.2 replaced three local user-first `scopeOrder` helpers with the canonical `compareByNameThenScope`, flipped `autoupdate.ts` iteration to project-first, and promoted MSG-GR-3 from a meta-assertion to an active two-axis AST detector (FunctionDeclaration + ArrayExpression visitors). The core change set is sound: the orchestrator iteration order is now consistently project-first, the catalog UAT adds `same-mp-both-scopes` regression locks for reinstall/update/import, and the lint rule's `RuleTester` companion exercises both axes.

However, the **scoping of the new MSG-GR-3 lint rule to `orchestrators/**/*.ts` left at least one production user-first `["user", "project"]` literal in `edge/handlers/plugin/import.ts:45` undetected**. That literal flows into `importClaudeSettings`'s `selectedScopes` array and determines the order of settings-loading and any top-level orphan diagnostic lines, contradicting the project-first contract the rest of the phase enforces. The lint rule cannot catch it (out of glob) and the catalog UAT's import fixtures all populate both scopes already, masking the drift.

Secondary findings include code duplication of `composeErrorWithCauseChain` across three orchestrator files (which the existing `sonarjs/no-identical-functions` rule should flag but does not because cross-file duplication is out of scope for that rule), stale block comments in `list.ts` that still cite the retired `["user", "project"]` iteration and the retired "No marketplaces configured." sentence, a no-op string transformation in the lint rule's URL builder, and `localeCompare` in `autoupdate.ts` that drops the catalog-mandated `sensitivity: 'base'` flag.

## Critical Issues

### CR-01: Edge handler still passes `["user", "project"]` to import orchestrator, contradicting project-first contract

**File:** `extensions/pi-claude-marketplace/edge/handlers/plugin/import.ts:45`
**Issue:** Phase 14.2 promoted MSG-GR-3 to an active AST check but scoped the rule (per `eslint.config.js:162`) to `extensions/pi-claude-marketplace/orchestrators/**/*.ts`. The bare-import edge handler still constructs `selectedScopes: parsed.scope === undefined ? ["user", "project"] : [parsed.scope]` and passes it to `importClaudeSettings`. That array drives `Promise.all(opts.selectedScopes.map(...))` in `execute.ts:898-903` and the iteration `for (const scopePlan of plan.scopes)` at `execute.ts:914`, which determines the order in which scopes are settings-loaded AND the insertion order of any top-level orphan diagnostic lines (`orphanDiagnosticLines` at `execute.ts:676-695`). Per-marketplace cascade blocks are independently sorted by `compareByNameThenScope` later, but orphan diagnostic lines bypass that sort. Result: when a bare `/claude:plugin import` produces scope-level diagnostics in both scopes, the user-scope diagnostic renders before the project-scope diagnostic, contradicting the same project-first contract this phase exists to enforce. The lint rule cannot detect it (out of glob); the catalog UAT's import fixtures all populate both scopes with `(added)` rows and so never exercise the orphan-diagnostic path.

The audit story Phase 14.2 tells (one canonical comparator, project-first everywhere) is incomplete by exactly this one literal.

**Fix:** Flip the literal to project-first and either (a) extend the MSG-GR-3 `files:` glob to include `edge/handlers/**/*.ts`, or (b) factor the scope-list into a shared helper in `shared/types.ts` (alongside the existing `SCOPES` constant at `shared/types.ts:20` -- which is itself user-first and would benefit from the same flip when audited for ordering semantics).

```ts
// extensions/pi-claude-marketplace/edge/handlers/plugin/import.ts:45
selectedScopes: parsed.scope === undefined ? ["project", "user"] : [parsed.scope],
```

And in `eslint.config.js` Block 1 (line 162):
```js
files: [
  "extensions/pi-claude-marketplace/orchestrators/**/*.ts",
  "extensions/pi-claude-marketplace/edge/handlers/**/*.ts",
],
```

(Note: `shared/types.ts:20` `SCOPES = ["user", "project"]` and `edge/completions/provider.ts:70` are also out of the lint rule's reach; audit them at the same time even if their current consumers don't propagate the order to user-visible output.)

### WR-04: `autoupdate.ts` `localeCompare` drops `sensitivity: 'base'` flag mandated by catalog

**File:** `extensions/pi-claude-marketplace/orchestrators/marketplace/autoupdate.ts:178`
**Issue:** `[...rows].sort((a, b) => a.name.localeCompare(b.name))` uses the default-sensitivity `localeCompare`. The catalog at `docs/output-catalog.md:79-81` and `:718` mandates `localeCompare` with `sensitivity: 'base'` (case-insensitive) for every marketplace sort surface. The comment at lines 170-177 acknowledges this is a per-call result list (not the marketplace-list surface) but still presents it as a sort to the user, so mixed-case marketplace names like `Alpha` and `beta` will render in a non-catalog order on the autoupdate result block while `marketplace list` would render them in the catalog order.
**Fix:**
```ts
const sorted = [...rows].sort((a, b) =>
  a.name.localeCompare(b.name, undefined, { sensitivity: "base" }),
);
```

### WR-05: MSG-GR-3 lint rule's URL builder regex is a no-op

**File:** `tests/lint-rules/msg-gr-3-per-scope.js:41-43`
**Issue:** `name.replace(/^msg-/, "msg-")` replaces the `msg-` prefix with itself -- equivalent to the bare `name`. The resulting docs URL becomes `https://github.com/.../docs/messaging-style-guide.md#msg-gr-3-per-scope`, which is not a real anchor in the style guide (real anchors are family-level, e.g. `#msg-gr-3`). The rule's `meta.docs.url` (auto-built by `RuleCreator`) will 404 when a developer clicks through from an IDE diagnostic.
**Fix:** Either drop the no-op replacement entirely (and accept the rule-name URL) or strip the trailing `-<slug>` to land on the family anchor:
```js
const createRule = ESLintUtils.RuleCreator(
  (name) =>
    `https://github.com/acolomba/pi-claude-marketplace/blob/main/docs/messaging-style-guide.md#${name.replace(/^msg-([a-z]+-\d+).*$/, "msg-$1")}`,
);
```

### WR-06: MSG-GR-3 `FunctionDeclaration` visitor is fragile to renames and structural drift

**File:** `tests/lint-rules/msg-gr-3-per-scope.js:114-122` (+ `isUserFirstScopeOrder` at 57-89)
**Issue:** The Pattern (a) detector only fires when the function is named exactly `scopeOrder` AND the body is exactly one `return` statement of the shape `return scope === "user" ? <num> : <num>` with the literal `"user"` on the RHS. Any of the following slips through:
  - Rename to `scopeRank` / `orderScope` / `compareScope` / etc.
  - Multi-statement body (`const u = "user"; return scope === u ? 0 : 1`)
  - Inverted comparator (`return scope === "project" ? 1 : 0`)
  - Symbol on the RHS (`const USER = "user"; return scope === USER ? 0 : 1`)
  - Arrow function or function expression instead of a `FunctionDeclaration`

The audit context establishes that the three known historical drift sites all happened to use the exact `scopeOrder` name with the exact one-line body, so this rule catches the historical pattern. But the rule advertises itself as "MSG-GR-3 active enforcement" and the next drift author may not match the historical shape.
**Fix:** At minimum, widen the visitor to `FunctionDeclaration | FunctionExpression | ArrowFunctionExpression` and match on any function whose body's `return` ConditionalExpression compares `scope === "user"` (regardless of identifier name) -- the "user-first numeric encoding" is the actual smell, not the function name. Document the intentional limits inline so future maintainers know what slips through. Alternatively, encode the contract as "no `scope === "user"` comparisons in orchestrator files outside of an explicit whitelist" and let the canonical comparator handle all ordering.

## Info

### IN-01: Two-element `ArrayExpression` visitor in lint rule could collide with unrelated literals

**File:** `tests/lint-rules/msg-gr-3-per-scope.js:127-141`
**Issue:** The Pattern (b) visitor fires on ANY 2-element `["user", "project"]` literal in any orchestrator file, regardless of context. If a future orchestrator legitimately needs a user-first iteration (e.g., a one-off enumeration in test fixtures imported into orchestrator code, or a debug helper), the only escape hatch is `// eslint-disable-line`. There is no annotation mechanism (`// MSG-GR-3-allow: ...`) for sanctioned local exemptions. Given the rule is new, this is unlikely to bite immediately, but it removes a degree of freedom that the previous (no-op) meta-assertion form preserved.
**Fix:** No code change required. Add a brief paragraph to the rule's `meta.docs.description` explaining that the rule has no allowlist mechanism and the canonical escape is an `eslint-disable-line` with an `MSG-GR-3` justification comment.

### IN-02: `renderPartition` helper in `marketplace/shared.ts` may be dead in the live tree

**File:** `extensions/pi-claude-marketplace/orchestrators/marketplace/shared.ts:179-199`
**Issue:** `renderPartition` is exported but the only consumers found via grep are in worktree snapshots (not the live `extensions/` tree). The Wave 1/2 cascade migration replaced partition-style rendering with `cascadeSummary` per CMC-26. If no live consumer remains, this helper is fallow alongside the legacy `Reinstalled:` / `Failed:` partition headings that Plan 13-02a-01 retired.
**Fix:** Audit the live tree for `renderPartition` consumers; remove the export if none remain. Same remediation pattern as WR-07.

### IN-03: Catalog UAT `examples.length >= 30` floor is a brittle smoke check

**File:** `tests/architecture/catalog-uat.test.ts:1745-1748`
**Issue:** The minimum-count assertion exists to catch silent loss of `<!-- catalog-state: -->` annotations after edits, but the floor is a magic number that drifts with every catalog addition. The 14.2-01 D-03 same-mp-both-scopes additions just bumped the actual count; the floor was not updated. Future additions silently lower the safety margin until the floor matches the actual count, at which point any deletion goes undetected for one revision.
**Fix:** Either pin the floor to the exact expected count (and update it as part of catalog-touching commits) or maintain a separate `EXPECTED_CATALOG_STATES` set in the test and assert set-equality against the parsed annotations. The set approach catches both additions and removals without a magic floor.
